# Product Proposal: FileSort Organizer (LIFA)

## 1. Описание

**FileSort Organizer** (рабочее название **LIFA — Local Intelligent File Architect**) — это настольное приложение для Windows (с планами на macOS/Linux), которое автоматически сортирует файлы в заданных папках, используя семантическое понимание содержимого, а не только расширения. Продукт работает полностью локально, обеспечивая конфиденциальность данных, и включает статический анализ безопасности для скриптов, предупреждая пользователя о потенциальных рисках.

## 2. Почему это нужно? (Проблема и боль)

Сегодня пользователи сталкиваются с «цифровым хаосом»: папка «Загрузки» и рабочий стол превращаются в свалку из документов, изображений, архивов и скриптов. Ручная сортировка отнимает время (до нескольких часов в месяц), а существующие автоматические инструменты имеют серьёзные ограничения:

| Тип решения                | Ограничения                                                                                   |
| -------------------------- | --------------------------------------------------------------------------------------------- |
| **Правила «если-то»**      | Работают только по расширению или имени, не понимают смысл файла (например, `отчет_2024.docx` и `счет_ivanov.pdf» оба относятся к работе, но правила не объединят их в одну папку). |
| **Облачные органайзеры**   | Требуют загрузки файлов на сервер, что нарушает конфиденциальность (медицинские, финансовые, корпоративные данные). |
| **Антивирусы**             | Проверяют только вредоносный код, но не классифицируют файлы для удобства хранения.           |
| **Поисковые системы (OS)** | Помогают найти файл, но не решают проблему организации.                                      |

Пользователь остаётся один на один с хаосом или тратит время на создание и поддержание структуры папок.

## 3. Целевая аудитория

- **Обычные пользователи Windows**, у которых папка «Загрузки» содержит сотни файлов, и им лень/некогда разбирать их.
- **Специалисты, работающие с разнотипными файлами**: разработчики (скрипты, документация), дизайнеры (изображения, проекты), аналитики (отчёты, данные).
- **Пользователи, заботящиеся о приватности**: фрилансеры, малый бизнес, юристы, врачи — все, кто не может позволить себе загрузку конфиденциальных файлов в облако.
- **Технически подкованные пользователи**, которые хотят автоматизировать рутину, но не умеют/не хотят писать сложные скрипты.

## 4. Решение: что делает PoC

PoC (Minimum Viable Product) реализует базовый сценарий «настроил и забыл»:

1. **Мониторинг папки в реальном времени**  
   Пользователь указывает одну или несколько папок для наблюдения (например, `C:\Users\Имя\Downloads`). Приложение использует `FileSystemWatcher` для отслеживания новых файлов.

2. **Извлечение текста и метаданных**  
   В зависимости от типа файла:
   - **Документы** (PDF, DOCX, XLSX, PPTX, TXT) — извлечение текста.
   - **Изображения** (JPEG, PNG и др.) — чтение EXIF-данных (дата, камера, GPS).
   - **Архивы** (ZIP, RAR) — список файлов внутри.
   - **Скрипты** (PY, PS1, BAT, SH) — получение исходного кода.
   - **Другие форматы** — анализ имени и расширения.

3. **Локальная семантическая классификация с помощью LLM**  
   Извлечённая информация передаётся локальной модели **Llama 3.2 3B** (через Ollama/Nexa). Модель:
   - Определяет категорию файла из предопределённого списка (документ, изображение, архив, скрипт, презентация, таблица, другое).
   - Предлагает имя целевой папки на языке пользователя (настраивается).
   - Для скриптов дополнительно проводит статический анализ (поиск подозрительных паттернов: `eval`, `base64`, сетевые вызовы, обфускация) и возвращает флаг риска.

4. **Принятие решения и перемещение**  
   - Если категория определена с высокой уверенностью, система создаёт (если нужно) соответствующую папку в целевой директории (например, `C:\Users\Имя\Sorted\Работа\Счета`) и перемещает файл.
   - Для скриптов с обнаруженными рисками пользователю показывается диалог с предупреждением и деталями — только после подтверждения выполняется перемещение.
   - Для остальных файлов можно настроить автоматическое перемещение или запрос подтверждения.

5. **Логирование и аудит**  
   Все действия записываются в локальные JSON-логи, доступные пользователю для просмотра и отката.

**Data Flow (уточнённый):**

```text
┌──────────────┐    ┌──────────────┐    ┌──────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│ FileSystem   │───▶│ Extractor     │───▶│ LLM Classifier   │───▶│ Decision Engine │───▶│ File Mover      │
│ Watcher      │    │ (текст, мета) │    │ (Llama 3.2)      │    │ (с учётом риска)│    │ (с подтвержд.)  │
└──────────────┘    └──────────────┘    └──────────────────┘    └─────────────────┘    └─────────────────┘
       │                    │                     │                      │                       │
       ▼                    ▼                     ▼                      ▼                       ▼
┌────────────────────────────────────────────────────────────────────────────────────────────────────┐
│                                          Локальные логи                                              │
└────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

## 5. Ключевые отличия от аналогов

- **Семантическая классификация** вместо правил по расширению. Модель понимает, что «договор_аренды.pdf» и «квитанция_оплаты.xlsx» относятся к финансовым документам, даже если в названиях нет общих слов.
- **Полная локальность** — никакие данные не покидают устройство. Пользователь не рискует конфиденциальностью.
- **Безопасность скриптов** — статический анализ и предупреждение перед любым действием. Это защищает от случайного запуска вредоносного кода.
- **Поддержка широкого спектра форматов** (не только документы, но и изображения, архивы, скрипты).
- **Гибкость** — пользователь может переопределять категории, задавать свои правила, отключать автоматическое перемещение.

## 6. Метрики успеха

### Продуктовые метрики (для оценки эффективности для пользователя)

| Метрика                              | Целевое значение (PoC)                           | Способ измерения                           |
| ------------------------------------ | ------------------------------------------------ | ------------------------------------------ |
| **Процент файлов, попавших в Others** | < 10% (чем меньше, тем лучше классификация)     | Подсчёт файлов, перемещённых в папку Others |
| **Время, сэкономленное пользователем** | Не менее 30 минут в неделю (оценка по опросу)   | Опрос пользователей после недели использования |
| **Количество отказов (Undo)**         | < 5% от общего числа перемещений                 | Логи действий и откатов                    |

### Технические метрики (производительность, надёжность)

| Метрика                               | Целевое значение (PoC)      | Способ измерения                        |
| ------------------------------------- | --------------------------- | --------------------------------------- |
| **Время обработки одного файла**      | < 5 секунд (для текстовых)  | Тайминг в логах                         |
| **Потребление RAM**                   | < 4 ГБ (при работе модели)  | Мониторинг системы                      |
| **Процент успешных перемещений**      | > 99% (без ошибок FS)       | Логи ошибок                             |
| **Доступность (Uptime)**               | > 99% (при активном сеансе) | Логи работы приложения                  |

### Агентские метрики (качество классификации)

| Метрика                  | Целевое значение              | Способ измерения                                 |
| ------------------------ | ----------------------------- | ------------------------------------------------ |
| **Accuracy**             | > 90% на тестовом наборе      | Сравнение с ручной разметкой (20-30 файлов)      |
| **Precision для скриптов** | 95% (минимум ложных срабатываний по рискам) | Статический анализ + ручная проверка |
| **Recall для скриптов**    | 90% (детекция реально опасных) | Тестовый набор вредоносных скриптов              |

## 7. Сценарии использования

### Основной сценарий (Happy Path)

1. Пользователь скачивает файлы в папку «Загрузки»: `отчет_2024.docx`, `IMG_1234.jpg`, `setup.exe` (инсталлятор).
2. FileSort Organizer автоматически определяет:
   - `отчет_2024.docx` → категория «Документ», папка `Документы/Работа`.
   - `IMG_1234.jpg` → категория «Изображение», папка `Изображения/Личные`.
   - `setup.exe` → категория «Исполняемый файл», но поскольку это не скрипт, просто перемещается в папку `Программы` (с подтверждением, если настроено).
3. Файлы перемещаются в соответствующие подпапки внутри `C:\Users\Имя\Organized`.
4. Пользователь замечает, что всё аккуратно разложено, экономит время на ручной сортировке.

### Edge-кейс: скрытый вредоносный скрипт

1. В папку «Загрузки» попадает файл `document.txt`, но внутри он содержит PowerShell-код с подозрительными вызовами.
2. Модуль извлечения читает текст, определяет, что это скрипт (по содержимому, а не по расширению).
3. LLM и статический анализатор находят паттерны: попытка скачать файл с внешнего URL, использование `Invoke-Expression`.
4. Система не перемещает файл автоматически, а показывает предупреждение:
   - «Обнаружен скрипт с подозрительными действиями: обращение к сети, eval. Переместить?»
5. Пользователь видит риск и принимает решение (например, удалить или оставить на карантине).

### Edge-кейс: неоднозначная классификация

1. Файл `проект_презентация.key` (Keynote) — расширение не поддерживается напрямую.
2. Извлечь текст не удаётся (бинарный формат).
3. LLM анализирует имя файла и предлагает категорию «Презентация» с низкой уверенностью (0.6).
4. Система может:
   - Переместить в папку `Другое/Презентации` (если настроен порог авто-действия).
   - Или запросить подтверждение у пользователя.

### Режим Dry Run

1. Пользователь хочет сначала посмотреть, что предложит система, без реального перемещения.
2. Включает режим предварительного просмотра.
3. Приложение обрабатывает все файлы в папке, но вместо перемещения выводит отчёт в интерфейсе:
   - `отчет_2024.docx` → `Документы/Работа`
   - `image.png` → `Изображения/Скриншоты`
4. Пользователь утверждает план, и система выполняет перемещение.

### Компоненты подробнее

| Компонент          | Технологии                                                                                             | Ответственность                                                                        |
| ------------------ | ------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------- |
| **Watcher**        | `watchdog` (кроссплатформенная обёртка над `FileSystemWatcher`)                                       | Отслеживание событий файловой системы (создание, изменение) и постановка в очередь.    |
| **Extractor**      | `python-docx`, `PyPDF2`, `openpyxl`, `python-pptx`, `PIL`, `zipfile`, `rarfile`, `pyarrow` (для парсинга кода) | Извлечение текстового содержимого и метаданных из файлов разных форматов.              |
| **Classifier**     | `Ollama` или `Nexa SDK` (Llama 3.2 3B) + статический анализатор (`ast`, `regex`)                     | Получение категории, имени папки, оценка безопасности скриптов.                        |
| **Decision Engine**| Встроен в Orchestrator                                                                                 | Применение порогов уверенности, пользовательских правил, запуск подтверждения.        |
| **File System Tool**| `shutil`, `pathlib`, `os`                                                                             | Физическое перемещение, создание папок, проверка прав доступа.                         |
| **Storage**        | `SQLite` + файловые JSON-логи                                                                          | Хранение настроек, истории операций, логов.                                            |
| **UI**             | `PyQt6` или `Tkinter` (для окон), `pystray` (для трея)                                                | Настройка, отображение логов, диалоги подтверждения.                                   |

## 9. Потенциальные сценарии использования (расширенные)

Помимо базовой сортировки, продукт может быть расширен до:

- **Интеграции с облачными хранилищами** (Google Drive, Dropbox) для синхронизации структуры (но без выгрузки содержимого).
- **Автоматического переименования** файлов на основе содержимого (например, «Счет_ООО_Ромашка_2025-03-06.pdf»).
- **Обучения собственной лёгкой модели** для ускорения классификации на слабых ПК (дистилляция Llama).
- **Поддержки плагинов** для пользовательских обработчиков (например, извлечение метаданных из специфических CAD-форматов).
- **Интеграции с антивирусным движком** (ClamAV) для углублённого анализа скриптов.

## 11. Риски и предположения

- **Производительность LLM на слабых ПК**: Llama 3.2 3B требует ~4 ГБ RAM и современный CPU. Для компьютеров без выделенной графики latency может превышать 5 секунд. Возможное решение: использовать квантованную версию (Q4) или более лёгкую модель (Phi-3-mini).
- **Точность классификации для некоторых форматов**: например, ZIP-архивы могут содержать смесь документов — модель может ошибаться. Потребуется иерархическая классификация или анализ метаданных архива.
- **Принятие пользователями**: необходимо обучение и понятный интерфейс. Важно объяснить, что данные остаются локальными.

## Заключение

**FileSort Organizer** решает реальную проблему цифрового хаоса, сохраняя приватность и добавляя интеллектуальный анализ безопасности. Благодаря локальной LLM и статическому анализу, продукт не только упрощает организацию файлов, но и защищает пользователя от случайного запуска вредоносных скриптов. Продуманная модель монетизации позволяет сделать базовую версию бесплатной, а Pro-функции — привлекательными для активных пользователей и бизнеса.

Следующий шаг — разработка MVP с фокусом на ключевой функционал и получение обратной связи от первых пользователей.
