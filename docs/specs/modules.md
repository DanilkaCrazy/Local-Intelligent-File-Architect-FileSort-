### 1. Retriever (кэш классификации)

Хотя в системе нет классического RAG, присутствует **кэш результатов классификации** по хешу файла. Функционально он играет роль «retriever» – избегает повторного вызова LLM для уже обработанных файлов.

- **Источники данных:**  
  - Локальная таблица SQLite: `cache(hash TEXT PRIMARY KEY, category TEXT, target_folder TEXT, confidence REAL, created TIMESTAMP)`.
  - Хеш вычисляется как SHA‑256 от содержимого файла (первые 1 МБ для больших файлов).
- **Индекс:**  
  - Первичный ключ по `hash`. Дополнительный индекс по `created` для очистки старых записей.
- **Поиск:**  
  - При получении нового файла вычисляется хеш → прямой запрос к SQLite.  
  - Если запись найдена и `created` не старше 30 дней → используется результат без вызова LLM.
- **Ограничения:**  
  - Не применяется для скриптов с уровнем риска `high` (всегда свежий анализ).  
  - Размер кэша ограничен 10 000 записей (автоматическая ротация).  
  - Кэш не синхронизируется между разными ПК (локальный).
- **Reranking (не применимо):** отсутствует, так как нет множества кандидатов.

### 2. Tools / APIs

#### 2.1. LLM API (Ollama / llama.cpp)

- **Контракт:**  
  `POST /api/generate`  
  Тело запроса:
  ```json
  {
    "model": "llama3.2:3b",
    "prompt": "<системный промпт>\n<file_content>...</file_content>",
    "stream": false,
    "temperature": 0.2,
    "max_tokens": 150
  }
  ```
  Ответ:
  ```json
  { "response": "{\"category\": \"document\", \"target_folder\": \"Финансы\", \"confidence\": 0.92, \"reason\": \"...\"}" }
  ```
- **Ошибки:**  
  - Таймаут (30 сек) → повтор до 3 раз с экспоненциальной задержкой.  
  - HTTP 5xx / ConnectionError → fallback на облегчённый режим (правила).  
  - Некорректный JSON в ответе → повтор с temperature=0.3 (максимум 2 раза), затем fallback.
- **Timeout:** 30 секунд на запрос. Общий таймаут на обработку файла – 45 сек (включая извлечение и анализ).
- **Side effects:** отсутствуют (модель не модифицирует внешнее состояние).
- **Защита:**  
  - Ограничение входного контекста до 4000 токенов.  
  - Валидация выходной JSON-схемы (используется `jsonschema`).  
  - Экранирование спецсимволов в `file_content` (чтобы не ломать JSON).

#### 2.2. File System API (через `shutil`, `os`, `pathlib`)

- **Контракт:**  
  - `shutil.move(src, dst)` – перемещение.  
  - `os.makedirs(dst_dir, exist_ok=True)` – создание целевой папки.  
  - `os.access(src, os.R_OK)` – проверка прав на чтение перед извлечением.
- **Ошибки:**  
  - `PermissionError` → логирование, файл остаётся на месте, уведомление пользователя.  
  - `FileNotFoundError` → исходный файл мог быть удалён; пропустить.  
  - `OSError` (диск полон, путь слишком длинный) → fallback с повтором через 5 сек (до 3 раз).
- **Timeout:** операции синхронные, но выполняются в отдельном потоке, чтобы не блокировать очередь. Таймаут не устанавливается (ОС сама решает).
- **Side effects:** физическое перемещение файла – необратимо (логируются исходный и целевой пути).
- **Защита:**  
  - Сначала копирование во временную папку (`tempfile.mkdtemp`), затем удаление исходника – атомарность через `shutil.move` (переименование на том же томе).  
  - Проверка свободного места перед копированием (минимально 2× размер файла).  
  - Блокировка конкурентного доступа через `fcntl` (Linux) или `msvcrt` (Windows) – опционально.

### 3. Memory / Context

- **Session state:**  
  В PoC нет долгоживущих сессий. Каждый файл обрабатывается изолированно. Однако для возможности отката (`undo`) хранится **история действий** в SQLite:  
  `history(id, timestamp, src_path, dst_path, category, user_confirmed)`.
- **Memory policy:**  
  - LLM не имеет persistent памяти между вызовами.  
  - Кэш классификации (см. Retriever) – исключение, но он не передаётся модели.  
  - Логи и метрики не содержат содержимого файлов.
- **Context budget:**  
  - Максимальная длина извлечённого текста, передаваемого в LLM – **4000 токенов** (оценка через `tiktoken` для модели).  
  - Если текст длиннее, обрезается: для документов – первые N страниц; для скриптов – первые 2000 символов + последние 500 (чтобы не потерять хвост с опасными вызовами).  
  - Системный промпт занимает ~500 токенов, остальное – содержимое файла.

### 4. Agent / Orchestrator

Оркестратор управляет последовательностью шагов для каждого файла. Реализован как конечный автомат (FSM) с поддержкой retry и fallback.

#### 4.1. Шаги выполнения (порядок)
1. **Extract** – извлечение текста/метаданных/кода.  
2. **SecurityScan** – только для скриптов (AST/regex).  
3. **Classify** – вызов LLM (или fallback на правила).  
4. **Decision** – применение порогов и пользовательских правил.  
5. **Confirm** – если требуется подтверждение, показать диалог.  
6. **Move** – выполнение перемещения.

#### 4.2. Правила переходов
- Если `Extract` вернул ошибку → переход к шагу `Classify` с флагом `low_confidence = true` и использованием только имени/расширения.  
- Если `SecurityScan` обнаружил `risk = high` → в `Decision` устанавливается `need_confirm = true` независимо от уверенности LLM.  
- Если `Classify` (LLM) не ответил после retry → переход на `ClassifyFallback` (правила).  
- После `Move` – переход в терминальное состояние `Done` или `Failed`.

#### 4.3. Stop condition
Обработка файла прекращается, когда:
- Файл успешно перемещён.  
- Пользователь явно отказался от перемещения (диалог подтверждения).  
- Произошла фатальная ошибка (например, диск полон, нет прав на целевую папку, и retry не помог).  
- Превышен общий таймаут обработки (45 сек).

#### 4.4. Retry / Fallback
| Компонент | Количество retry | Задержка | Fallback |
|-----------|------------------|----------|----------|
| LLM запрос | 3 | 1, 2, 4 сек | Правила (расширение/имя) |
| Перемещение | 3 | 1 сек | Пропуск файла, лог ошибки |
| Извлечение (например, PDF повреждён) | 1 | 0.5 сек | Использовать только имя/расширение |
| Создание папки | 2 | 1 сек | Попытка создать в альтернативной папке (например, `_failed`) |

- Все fallback-решения логируются с указанием причины.

### 5. Serving / Config

#### 5.1. Запуск
- **Точка входа:** `main.py`  
- **Последовательность:**  
  1. Чтение `config.yaml` и `config.local.yaml` (переопределение).  
  2. Инициализация SQLite (таблицы кэша, правил, истории).  
  3. Бенчмаркинг доступных LLM (если `model: auto`).  
  4. Запуск `FileSystemWatcher` для указанных папок.  
  5. Запуск фоновых воркеров (пул потоков, размер = CPU ядра).  
  6. Отображение трей-иконки.

#### 5.2. Конфигурация (пример `config.yaml`)
```yaml
watched_folders:
  - "C:\\Users\\%USERNAME%\\Downloads"
  - "C:\\Users\\%USERNAME%\\Desktop"
target_base: "C:\\Users\\%USERNAME%\\Organized"
model: "auto"   # или "llama3.2:3b", "phi3:mini", "gemma2:2b"
fallback_mode: "rules"   # "llm" или "rules"
dry_run: false
confidence_threshold: 0.7
confirm_scripts: true
auto_move_non_scripts: true
max_file_size_mb: 1024
llm_timeout_sec: 30
log_level: "INFO"
```

#### 5.3. Секреты
- В PoC нет внешних API-ключей (всё локально).  
- При будущей интеграции с облачными сервисами (опционально) секреты будут храниться в системном хранилище (Windows Credential Manager, macOS Keychain).  
- Файл конфигурации не содержит паролей.

#### 5.4. Версии моделей
- При запуске с `model: auto` система выполняет бенчмаркинг:  
  - Загружает каждую из предустановленных моделей (список в `models_shortlist.yaml`).  
  - Прогоняет тестовый набор из 5 файлов (разных типов).  
  - Измеряет среднюю латентность и точность (сравнение с эталонными категориями).  
  - Выбирает модель с максимальным `accuracy / latency` (приоритет точности).  
- Результат сохраняется в `config.local.yaml`.  
- Пользователь может вручную указать модель в конфиге.

### 6. Observability / Evals

#### 6.1. Метрики (собираются и экспортируются через `prometheus_client` или логи)
| Название | Тип | Описание |
|----------|-----|----------|
| `files_processed_total` | Counter | Всего обработано (по типам: document, image, script, other) |
| `files_moved_total` | Counter | Успешно перемещено |
| `files_failed_total` | Counter | Не удалось переместить (с меткой причины) |
| `files_skipped_total` | Counter | Пропущено пользователем или из-за политики |
| `llm_inference_duration_seconds` | Histogram | Время одного запроса к LLM (бакеты 0.5, 1, 2, 5, 10) |
| `llm_errors_total` | Counter | Ошибки LLM (таймаут, некорректный JSON) |
| `queue_length` | Gauge | Текущая длина очереди файлов |
| `fallback_triggered_total` | Counter | Срабатывание fallback-режима (с меткой причины) |
| `memory_usage_mb` | Gauge | Потребление RAM процессом |

#### 6.2. Логи
- **Формат:** JSONL (одна строка – одно событие).  
- **Ротация:** 10 файлов по 10 МБ, хранить 30 дней.  
- **Уровни:** DEBUG, INFO, WARNING, ERROR.  
- **Обязательные поля в каждом событии:** `timestamp`, `level`, `module`, `file_hash` (если применимо).  
- **Что НЕ логируется:** содержимое файла, извлечённый текст, полный путь (только относительный или имя файла), пароли.  

Пример строки лога:
```json
{"timestamp":"2025-03-23T10:15:30Z","level":"INFO","module":"decision","file_hash":"sha256:abcd...","action":"move","src":"report.pdf","dst":"Organized/Finance/report.pdf","confidence":0.92}
```

#### 6.3. Трейсы (опционально для PoC)
- Для каждого файла генерируется уникальный `correlation_id` (UUID).  
- Он передаётся через все модули (через `logging.LoggerAdapter`).  
- Позволяет восстановить полную цепочку обработки в логах.  
- В PoC отдельный сборщик трейсов (Jaeger) не используется, но формат логов совместим с OpenTelemetry.

#### 6.4. Проверки качества (Evals)
- **При установке:** автоматический тест на 50 репрезентативных файлах (размечены вручную). Вычисляется accuracy, precision, recall. Если accuracy < 85% – пользователю предлагается выбрать другую модель или перейти в облегчённый режим.  
- **Периодические (еженедельно):** сбор анонимной статистики (только с согласия пользователя) – какие категории ошибались, частота fallback’ов. Отправка на сервер разработчика (опционально) для улучшения модели.  
- **Ручная обратная связь:** в уведомлении после перемещения есть кнопка «Неправильно». При нажатии пользователь может указать правильную категорию – это сохраняется в локальной таблице `feedback` и используется для дообучения правил.

#### 6.5. Инфраструктурные проверки (health)
- **Health‑check LLM:** раз в 30 секунд отправляется тестовый запрос (промпт «ping»). При трёх последовательных неудачах система переключается в облегчённый режим и показывает уведомление.  
- **Health‑check диска:** проверка свободного места на целевом диске перед любым перемещением (< 100 МБ – остановка обработки до освобождения места).  
- **Health‑check очереди:** если длина очереди превышает 1000, новые файлы не ставятся, а логируются как пропущенные (с уведомлением пользователя).

## Дополнительные замечания по трекам

### Механизмы проверки и коррекции галлюцинаций LLM

1. **Структурная валидация выхода**  
   - LLM обязана отвечать строгим JSON с фиксированной схемой (`category`, `target_folder`, `confidence`, `reason`).  
   - Если парсинг JSON не удался или поле `confidence` < 0.3 — ответ считается галлюцинацией, выполняется **повторный запрос с temperature=0.1** (максимум 2 раза).  
   - Если повтор не помог — **fallback на облегчённый режим (правила)**.

2. **Перекрёстная проверка с правилами**  
   - Для скриптов: если LLM классифицировала файл как «документ» (низкий риск), но статический анализатор нашёл подозрительные паттерны — приоритет у **статистического анализа**, и будет показано предупреждение.  
   - Для известных расширений (например, `.exe`, `.msi`): если LLM предлагает категорию «изображение» — это явная галлюцинация; система игнорирует ответ LLM и использует правила.

3. **Логирование неопределённости**  
   - В лог записывается не только итоговая категория, но и «уверенность» (confidence), а также причина выбора (из LLM или из правил).  
   - Если confidence < 0.6, в лог добавляется флаг `low_confidence: true`.

4. **Обратная связь от пользователя**  
   - После каждого перемещения (в уведомлении) есть кнопка «Неправильно». При нажатии пользователь может указать правильную категорию.  
   - Эти данные сохраняются в таблице `feedback` и **периодически (раз в неделю)** автоматически анализируются: если для определённого типа файлов LLM часто ошибается, система предлагает пользователю либо скорректировать правила, либо переключиться на другую модель.

5. **Тестовый прогон при установке (Evals)**  
   - На тестовом наборе из 50 файлов (с ручной разметкой) вычисляется **accuracy**. Если accuracy < 85% — система предупреждает пользователя и предлагает:  
     а) выбрать другую модель (из доступных),  
     б) переключиться в облегчённый режим (правила),  
     в) продолжить, но с пониженным доверием (авто-перемещение только при confidence > 0.9).

### Агентский трек (качество LLM, защита)
- В спецификациях сделан акцент на:  
  - Выбор модели по метрикам (accuracy/latency).  
  - Контроль контекста (обрезание до 4000 токенов).  
  - Валидация JSON-ответа и повторные запросы при ошибках.  
  - Статический анализ скриптов как «второе мнение».  
  - Обязательное подтверждение для скриптов (даже если LLM сказала «безопасно»).  
- Для оценки качества введены `Evals` при установке и обратная связь от пользователя.

### Инфраструктурный трек (стабильность, мониторинг)
- В спецификациях проработаны:  
  - Retry и fallback для каждого внешнего вызова (LLM, файловая система).  
  - Метрики, логи, health‑checks.  
  - Управление ресурсами (ограничение размера файла, очереди, ротация логов).  
  - Защита от конкурентного доступа к файлам (временные копии).  
  - Автоматический fallback на облегчённый режим при недоступности LLM.  
- Коммитмент по latency и reliability (см. раздел системных требований).
